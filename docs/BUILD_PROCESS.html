Executive Overview

What we built and why it matters

What Deeter Intelligence Is

Deeter Intelligence is a live, production-deployed trading desk AI co-pilot — built as a direct response to the architecture challenge in the Head of Applied AI & Trading Systems job description at Deeter Analytics. Instead of submitting a theoretical write-up, we built the system and deployed it.

The JD asked candidates to describe how they would architect a real-time news filter for a trading desk. This is that system — working, deployed, and instrumented with real APIs. Every architectural decision mirrors what a production firm would actually build at $500M AUM scale.

◆

The demo answers the question directly: not "here is how I would build it" but "here is the system, live, handling real data, making real AI decisions." The GitHub repo is the application. The Vercel URL is the proof.

Most candidates submit a PDF or a GitHub with boilerplate. This submission is a live system that demonstrates every capability named in the JD: real-time data pipelines, multi-model AI consensus, voice interface, entity relationship mapping, and portfolio exposure calculation. The LLM Council

Multi-model consensus for borderline signals

How the Council Works

Single-model scoring has a well-known failure mode: confident-but-wrong. An article scoring 6/10 on relevance is genuinely ambiguous — it might be actionable or it might be noise. The cost of misclassification in either direction is real. The LLM Council runs borderline articles through three independent models before committing to a final score.

◆

The Council demonstrates production-grade AI thinking. It is not a gimmick. It directly addresses the hallucination awareness and multi-model orchestration capabilities called out in the JD.

Stage 1: Vote Each model independently scores the article without seeing other scores. Prevents anchoring bias. Claude, GPT-4o, Gemini Pro 3 independent votes: relevance + sentiment + reasoning
Stage 2: Review Each model receives all three Stage 1 votes and can revise its score with justification. Same 3 models Revised votes with documented reasoning shifts
Stage 3: Synthesize Chairman model (Claude) receives all votes and reasoning, synthesizes a final score with confidence level. Claude (Chairman) finalRelevance, finalSentiment, confidence, synthesis
Fault Tolerance

The Council uses Promise.allSettled throughout. If any model's API key is invalid or a request times out, that model's vote is excluded from synthesis. The Council proceeds with the available votes and downgrades the confidence level accordingly. It only fails if all three models fail simultaneously.

JARVIS Voice Layer

Bidirectional voice — the Iron Man interface

What JARVIS Does

JARVIS is the voice interface layer. It handles two directions: input (Web Speech API push-to-talk, routed to the chat AI) and output (ElevenLabs TTS for responses and proactive alerts). The design goal is a sharp desk analyst on call — not a chatbot that reads responses in a robotic voice.

The Greeting Problem — and How We Solved It

Browser autoplay policy blocks audio triggered by async code after a network fetch. A naive implementation would call ElevenLabs, await the response, then try to play audio — burning the user gesture context and getting silently blocked. The solution is a prefetch pattern.

  • 1 On mount: fetch the greeting audio from ElevenLabs in the background, cache the blob URL in a ref
  • 2 First mic click: inside the user gesture, play from the cached blob URL synchronously — no async gap, no autoplay block
  • 3 Fallback: if ElevenLabs wasn't preloaded, fall back to browser SpeechSynthesis with no error thrown
  • 4 Second mic click: starts recognition normally — first click was greeting only
Voice Recognition — Browser Compatibility
BrowserSpeech RecognitionElevenLabs TTSExperience
Chrome / Edge Full support (webkitSpeechRecognition) Full support Complete JARVIS experience
Safari / Firefox Not supported Full support TTS output only, mic button dimmed with "Voice requires Chrome or Edge" hint
◆

The voiceSupported state is detected client-side via useEffect to avoid SSR hydration mismatch. Unsupported browsers get a graceful degraded state — not a silent broken button.

Proactive Alerts

When a scored article hits relevance 9 or 10, JARVIS fires a proactive spoken alert without waiting for the trader to ask. The alert text is structured: ticker, headline summary, relevance score. Requires prior user activation (the mic click serves as the activation event).

Every demo failure mode, anticipated and handled

A demo that breaks in front of a hiring manager is worse than no demo. Every known failure mode was identified, reproduced, and resolved before deployment. This section documents the reliability work — not because it was asked for, but because production systems require it.

Deployment

Infrastructure, environment variables, and Vercel configuration

Production Infrastructure

Vercel handles all infrastructure. Next.js App Router routes are automatically deployed as Vercel Serverless Functions. Static assets are CDN-distributed globally. The Finnhub WebSocket runs client-side. All sensitive keys are injected as Vercel environment variables — never in the repo.
